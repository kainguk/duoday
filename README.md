## 사용자 가치 3줄 요약

- 하루 1개의 가벼운 질문으로, 바쁜 날에도 서로의 감정과 생각을 기록할 수 있습니다.
- 데이트 기록(사진/장소/감정 태그)을 타임라인으로 축적해 관계의 맥락을 오래 보존합니다.
- 쌓인 기록을 책으로 묶어 디지털 추억을 실제 소장 가능한 결과물로 전환합니다.

## 콘텐츠 기획 의도

DuoDay는 "특별한 날만 기록하는 서비스"가 아니라, 평범한 하루도 의미 있게 남기는 것을 목표로 합니다.
사진만으로 남기기 어려운 감정과 대화를 짧은 질문/답변으로 함께 저장해, 시간이 지난 뒤에도 관계의 맥락을 다시 읽을 수 있게 설계했습니다.

- **왜 질문형 콘텐츠인가**: 일기보다 작성 부담이 낮고, 서로의 생각 차이를 자연스럽게 드러낼 수 있음
- **왜 타임라인 구조인가**: 기록을 단건으로 소비하지 않고, 시간의 흐름 속에서 함께 회고하게 만들기 위함
- **왜 책 기능을 붙였는가**: 디지털 기록을 오프라인 결과물로 전환해 장기 소장 가치를 만들기 위함

핵심은 기능 자체보다, "**둘이 함께 쌓은 시간**"이 서비스 안에서 점점 선명해지게 만드는 것입니다.

## 서비스 소개

- **한 줄 소개**: 커플의 일상 콘텐츠(질문 답변 + 데이트 기록)를 축적하고 책으로 미리보기/주문/익스포트하는 웹앱
- **타겟 사용자**: 일상을 함께 기록하고 나중에 책으로 보관하고 싶은 커플
- **핵심 구조**: 콘텐츠가 본체(Lv1), 책 주문은 부가 기능(Lv2), 주문 데이터 익스포트(Lv3)

## 사용 흐름 (콘텐츠 관점)

1. **홈**: D+일수/시작일과 주요 지표를 통해 "지금까지의 시간"을 직관적으로 확인
2. **오늘의 질문**: 하루 1개 질문에 각자 답하면서 대화의 재료를 확보
3. **데이트 타임라인**: 사진, 장소, 감정 태그를 함께 기록해 사건의 맥락을 보존
4. **대시보드**: 많이 남긴 감정/함께 답변한 질문 수로 관계 패턴을 회고
5. **책 만들기**: 특정 기간의 질문/데이트를 묶어 소장 가능한 결과물로 정리

## 실행 방법 (Docker)

```bash
# 저장소 클론
git clone <repo-url>

# 저장소 진입
cd duoday-main

# 환경 변수 준비
cp .env.example .env

# 실행
docker compose up --build

# 접속
http://localhost:3000
```

- DB/업로드 볼륨: `duoday-db`, `duoday-uploads`
- 최초 실행 시 마이그레이션 + 시드가 자동 반영되어 로그인 없이 확인 가능한 더미 데이터가 생성됩니다.
- 기본 더미 질문 답변 분포(최근 데이터): 둘 다 미작성 1개 / 지원만 작성 1개 / 도윤만 작성 1개 / 나머지 둘 다 작성
- 위 분포를 의도적으로 유지한 이유(테스트 시나리오):
  - 로그인/비로그인 모드에서 답변 노출 조건이 정확히 동작하는지 빠르게 검증하기 위함
  - **지원만 작성**: 도윤 계정 로그인 시, 본인 미작성 상태에서 지원 답변이 가려지는지 확인
  - **도윤만 작성**: 지원 계정 로그인 시, 본인 미작성 상태에서 도윤 답변이 가려지는지 확인
  - **둘 다 미작성** 케이스: 양쪽 모두 작성 UI가 정상 노출되는지 확인
  - **둘 다 작성** 케이스: 저장 후 상호 답변 노출과 수정/삭제 후 뷰 모드 전환이 자연스러운지 확인

### 포트 변경

`docker-compose.yml`의 `ports`를 수정하세요.

```yml
ports:
  - "3100:3000"
```

그러면 `http://localhost:3100`으로 접속합니다.

## 완성한 레벨

| 레벨 | 구현 상태 | 설명 |
|---|---|---|
| Lv1 | 완료 | 오늘의 질문/지난 질문, 데이트 기록 CRUD, 다중 사진 + 대표사진, 감정 태그, 대시보드 |
| Lv2 | 완료 | 책 주문 생성/조회/상태 관리, 책 미리보기(표지/내지/마지막) |
| Lv3 | 완료 | 주문 1건 ZIP 익스포트(`order.json`, 콘텐츠 JSON, 사진 원본) |

## 주요 기능

### Lv1: 콘텐츠 서비스

- 오늘의 질문 자동 배정(라운드로빈), 답변 작성/수정/삭제
- 로그인 모드/비로그인 모드 분리, 답변 공개 조건 제어
- 데이트 기록 생성/조회/수정/삭제
- 다중 사진 업로드 + 대표사진 지정
- 감정 태그/베스트 표시, 필터링, 월별/태그 통계 대시보드
- 홈 히어로 메타에 `D+N`과 연애 시작일(예: `시작일 2024.09.14`) 표시
- 통계 카드의 "함께 답변한 질문"은 최근 8개 질문 중 두 사람이 모두 답변한 질문 수를 기준으로 집계

### Lv2: 책 주문 기능

- 기간/표지색/제목/포함 콘텐츠 선택 후 주문 생성
- 주문 상세/목록/상태 변경
- 상태 흐름: `pending`, `processing`, `completed`, `cancelled` (테스트용 되돌리기 포함)
- 책 미리보기(제본 여백, 내지 분할, 마지막 페이지 문구)

### Lv3: 주문 데이터 익스포트

- `/api/export/[id]`에서 주문 단위 ZIP 생성
- 포함 파일:
  - `order.json`
  - `content/questions.json`
  - `content/dates.json`
  - `content/photos/<dateId>/*`
  - `README.txt`

## 기술 스택 / 아키텍처

- **프론트/백엔드**: Next.js 14 App Router + React 18 + TypeScript
- **DB**: SQLite (`better-sqlite3`)
- **스타일**: Tailwind CSS
- **유틸**: `zod`, `nanoid`, `jszip`
- **배포/실행**: Docker, docker-compose (standalone build)

스택 선택 이유:

- **Next.js(App Router) + TypeScript**: 화면/서버/API를 한 저장소에서 빠르게 반복 개발하고, 타입 기반으로 요구사항 변경에 안정적으로 대응하기 위해 선택했습니다.
- **SQLite**: 과제 요구사항상 단일 서비스 독립 실행이 중요해, 별도 DB 인프라 없이 Docker에서 즉시 기동 가능한 구성을 우선했습니다.
- **Tailwind CSS**: 짧은 개발 기간에 화면 수정 요청을 빠르게 반영하고, 컴포넌트별 스타일 일관성을 유지하기 위해 사용했습니다.
- **Docker Compose**: 심사자 환경에서 실행 편차를 줄이고 `1회 명령 실행` 조건을 만족하기 위해 채택했습니다.

## 설계 의도

- **아이디어 선택 이유**: 커플 도메인은 기록 데이터가 자연스럽게 누적되고, 책 주문으로 확장하기 쉬움
- **사업적 가능성**: 구독형 기록 서비스 + 시즌/기념일별 책 주문으로 LTV 확장 가능
- **콘텐츠 운영 관점**:
  - 질문은 "깊은 글쓰기"가 아니라 "짧지만 자주 쓰는 습관"을 만들도록 난이도를 낮춤
  - 데이트 기록은 감정 태그를 함께 저장해 나중에 "무슨 일을 했나"뿐 아니라 "어떤 기분이었나"까지 복원
  - 통계는 단순 활동량보다 "함께 남긴 기록"을 강조해 관계 중심 지표로 해석 가능하게 구성
- **추가하고 싶은 기능**:
  - 실제 계정 인증 및 권한 모델 고도화
  - 책 레이아웃 템플릿 선택/폰트 옵션
  - 사진 자동 보정/표지 추천
